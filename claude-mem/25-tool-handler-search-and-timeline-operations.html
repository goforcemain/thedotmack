工具处理器：搜索与时间线操作

18 分钟等级: 进阶

MCP 服务器暴露了一组用于搜索记忆和检索按时间顺序排列的上下文的工具。这些工具遵循精心设计的 3 层工作流，通过从广泛索引到详细内容的逐步过滤来实现高效的 token 使用。工具处理程序系统通过统一的搜索基础设施，在 Claude Desktop 的 MCP 协议与 Worker Service 的 HTTP API 之间架起桥梁。

架构概览

工具处理程序系统作为 Claude Desktop 的 MCP 协议与 Worker Service 的 HTTP API 之间的桥梁运行。MCP 服务器作为一个 stdio 进程运行，而 Worker Service 提供一个处理实际搜索和时间线操作的 HTTP 服务器。

MCP 服务器在 src/servers/mcp-server.ts 中定义工具，并通过 TOOL_ENDPOINT_MAP 将请求路由到 Worker HTTP 端点。每个工具处理程序使用 GET 或 POST 方法调用 Worker API，后者随后委托给 SearchManager 进行处理。

来源：src/servers/mcp-server.ts，src/services/worker/http/routes/SearchRoutes.ts

3 层工作流模式

记忆操作的核心设计原则是 3 层工作流，与立即获取完整详情相比，这提供了大约 10 倍的 token 节省。

此工作流通过 __IMPORTANT 工具强制执行，该工具作为正确使用模式的文档和指导。该工作流确保用户在请求完整内容之前逐步过滤结果。

来源：src/servers/mcp-server.ts

搜索工具处理程序

search 工具是记忆发现的主要入口点。它支持语义向量搜索和直接 SQL 过滤，并具有自动回退策略。

搜索执行流程

搜索操作支持两种不同的执行路径：

  1. 仅过滤路径：当未提供查询文本时，系统使用直接 SQLite 过滤。此路径支持日期范围过滤、类型过滤和其他基于元数据的查询，而无需语义搜索功能。这对于按时间段或文档类型浏览特别有用。

  2. 语义搜索路径：当提供了查询文本且 ChromaDB 可用时，系统执行基于向量的语义搜索。此路径查询 Chroma 向量数据库以查找语义相似的文档，然后按时间范围（90 天窗口）过滤结果，并从 SQLite 补充完整详情。

来源：src/services/worker/SearchManager.ts

搜索结果结构

搜索结果按文档类型组织，并包含用于识别和过滤的基本元数据：

每个结果包括：ID、标题、时间戳、created_at_epoch、类型/元数据字段、文件关联和 token 估计值。结果按日期和文件分组以便高效浏览。

来源：src/services/worker/SearchManager.ts

搜索操作自动为语义搜索结果应用 90 天的近期过滤器。这通过专注于最近的活动来平衡相关性与性能，同时仍允许仅过滤查询访问整个时间范围内的历史数据。

时间线工具处理程序

timeline 工具提供围绕特定观察记录、会话或时间戳的按时间顺序排列的上下文。它支持基于锚点和基于查询的模式来检索周围的上下文。

时间线操作模式

时间线处理程序支持两种不同的操作模式：

  1. 基于查询的模式：当提供 query 参数时，系统首先搜索匹配的观察记录，然后使用顶部结果作为锚点。这对于在不了解确切 ID 的情况下探索特定主题周围的上下文很有用。

  2. 基于锚点的模式：当提供 anchor 参数时，系统直接检索围绕该锚点的时间线数据。锚点可以是：

    • 数字观察 ID（例如：123）
    • 会话 ID（例如：S123 或 #S123）
    • ISO 时间戳（例如：2025-01-15T10:30:00Z）

来源：src/services/worker/SearchManager.ts

时间线参数

参数类型必填描述
anchornumber/string是*观察 ID、会话 ID (S#) 或 ISO 时间戳
querystring是*搜索查询以自动查找锚点
depth_beforenumber否锚点之前的项目数（默认：10）
depth_afternumber否锚点之后的项目数（默认：10）
projectstring否按项目名称过滤

*必须提供 anchor 或 query 中的一个

SearchManager.timeline() 中的验证逻辑确保用户提供两个必需参数中的一个，并为无效的组合返回清晰的错误消息。

来源：src/servers/mcp-server.ts，src/services/worker/SearchManager.ts

时间线深度过滤

深度过滤机制确保时间线结果保持在锚点周围的请求窗口内：

TimelineBuilder.filterByDepth() 方法在排序的时间线中定位锚点项目，然后应用 depth_before 和 depth_after 约束来过滤结果集。这确保了在不加载不必要数据的情况下进行高效检索。

来源：src/services/worker/search/TimelineBuilder.ts

时间线结果格式化

时间线结果格式化为包含日期和文件分组的结构化 markdown：

TimelineService.formatTimeline() 方法处理格式化，为不同的观察类型应用图标，按天分组，并在相关处显示文件上下文。

来源：src/services/worker/TimelineService.ts

时间线请求在单个查询中自动检索观察记录、会话和提示词，然后按时间顺序合并和排序。这种统一方法提供了围绕锚点的完整上下文，而无需多个单独的查询。

错误处理和回退

搜索和时间线处理程序实现了强大的错误处理机制，并具有优雅的回退机制：

ChromaDB 不可用

当 ChromaDB 未初始化或在语义搜索期间失败时：

  1. 系统记录一条调试消息，指示语义搜索不可用
  2. 返回一条有用的错误消息，其中包含 uv/Python 的设置说明
  3. 仍然允许仅过滤搜索（日期范围、类型、文件）运行
  4. 提供指向安装文档的链接

来源：src/services/worker/SearchManager.ts，src/services/worker/SearchManager.ts

无效参数

时间线处理程序验证输入参数并返回特定的错误消息：

  • 同时缺少 anchor 和 query："必须提供 'anchor' 或 'query' 参数"
  • 同时提供了 anchor 和 query："不能同时提供 'anchor' 和 'query' 参数"
  • 无效的观察 ID："未找到观察 #X"
  • 无效的会话 ID："未找到会话 #X"
  • 无效的时间戳："无效的时间戳: X"
  • 无效的锚点类型："无效的锚点：必须是观察 ID（数字）、会话 ID（例如 'S123'）或 ISO 时间戳"

来源：src/services/worker/SearchManager.ts

空结果

当未找到结果时：

  • 搜索：返回"未找到匹配 'query' 的结果"消息
  • 时间线（查询模式）：返回"未找到匹配 'query' 的观察记录。尝试不同的搜索查询。"
  • 时间线（锚点模式）：返回"在锚点周围未找到上下文（X 条记录前，Y 条记录后）"

来源：src/services/worker/SearchManager.ts，src/services/worker/SearchManager.ts

性能优化

搜索和时间线处理程序实现了几种性能优化：

  1. 批处理操作：get_observations 工具需要批处理获取（2+ 个 ID）以分摊 HTTP 请求开销
  2. 近期窗口：语义搜索自动过滤到 90 天窗口以减少结果处理
  3. 深度过滤：时间线操作尽可能在数据库级别应用深度约束
  4. 基于索引的首遍：搜索在完整详情之前返回轻量级索引结果（ID 和元数据）
  5. 连接重用：到 Worker API 的 HTTP 连接在请求之间重用

来源：src/services/worker/search/types.ts，src/services/worker/SearchManager.ts
